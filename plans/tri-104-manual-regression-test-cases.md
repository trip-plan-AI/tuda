# TRI-104 — Полный regression-набор для ручного тестирования

## 1. Область изменений

Набор покрывает ключевые сценарии синхронизации AI ↔ Planner из ветки `feature/TRI-104-ai-planner-interaction`:
- переходы AI → Planner через `applyTripId` и `draftMessageId`
- переход Planner → AI через «Редактировать с AI»
- поведение `applyPlanToCurrentTrip` для первого и повторного применения
- unified-модалка конфликтов на страницах AI, Landing и Tour Detail
- защита от потери несохранённого маршрута и от повторной обработки query-параметров

## 2. Общие предусловия тестового окружения

1. Запущены web и api окружения проекта.
2. Есть тестовый пользователь с правами входа.
3. Для части кейсов нужен минимум один сохранённый маршрут с 2+ точками.
4. Для части кейсов нужен AI-чат с ответом, содержащим route plan.
5. Открыт DevTools (вкладки Network и Console) для валидации запросов/ошибок.

## 4. Дополнительные негативные проверки

1. При обрыве сети во время перехода AI → Planner не должно быть «вечного лоадера» в AI.
2. При 403/401 API в сценарии apply/openOrCreateSession пользователь видит контролируемую ошибку, а не падение страницы.
3. При ручном удалении query-параметров из URL в процессе загрузки Planner не должен уходить в цикл.

## 5. Чек-лист приёмки прогона

- [ ] Все P0-кейсы прошли без блокеров
- [ ] Нет silent overwrite маршрута после первого apply из AI
- [ ] Конфликтные модалки открываются строго в нужных условиях
- [ ] Все 4 действия модалки работают в AI, Landing, Tour Detail
- [ ] Переход Planner → AI сохраняет маршрут и открывает корректную сессию
- [ ] Query `applyTripId`/`draftMessageId` корректно очищаются после применения
- [ ] Нет повторного авто-применения одного и того же ключа trip-draft
- [ ] Карта и точки соответствуют целевому trip после подмены
- [ ] Ошибки API обрабатываются без критического краша UI
- [ ] Мобильный сценарий не имеет функциональных регрессий

## 6. Рекомендуемый порядок прогона

1. Сначала полностью закрыть P0 кейсы `TRI104-MAN-001..015`.
2. Затем пройти унификацию модалок `TRI104-MAN-018..019`.
3. Завершить edge/deep-link кейсами `TRI104-MAN-021..024`.
4. После прогона заполнить чек-лист приёмки и зафиксировать найденные дефекты по ID кейса.

## 7. Smoke-чеклист поверх regression (быстрый прогон 10-12 шагов)

Цель smoke: быстро подтвердить, что критичный пользовательский флоу TRI-104 не сломан после свежих изменений/деплоя.

1. Авторизоваться и открыть AI Assistant.
2. Убедиться, что список чатов загрузился и активный чат доступен.
3. Отправить короткий запрос в AI и дождаться ответа со skeleton без UI-падений.
4. Нажать «Применить план» в сессии без привязанного `tripId`.
5. Проверить, что показывается успешная синхронизация и чат получает привязку к trip.
6. Нажать «Открыть Planner» из AI и убедиться, что маршрут открылся корректно.
7. Проверить, что после применения в Planner URL очищен от `applyTripId` и `draftMessageId`.
8. В Planner нажать «Редактировать с AI» и убедиться, что открылся релевантный чат по текущему trip.
9. Сгенерировать в этом же чате новую версию плана и открыть Planner повторно.
10. Проверить появление модалки конфликта для same_route или different_route по ситуации.
11. В модалке нажать «Заменить без сохранения» и убедиться, что переход/замена выполняются.
12. Повторно вызвать модалку и проверить «Отмена» и «Перейти в планнер посмотреть текущий».

### Критерий успеха smoke

- Нет блокирующих ошибок JavaScript и белых экранов.
- Основные переходы AI ↔ Planner работают.
- Модалка конфликтов открывается и выполняет действия.
- Однократная обработка `applyTripId` сохраняется.
