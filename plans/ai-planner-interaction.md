# Анализ плана интеграции AI-Assistant и Planner

## 1. Соответствие документации (PRD, Dev Guide и Backend-Only Policy)

Ваш план отлично соотносится с архитектурой продукта и требованиями.

- В БД уже предусмотрена таблица `ai_sessions` с полем `tripId` (связь сессии с маршрутом один-к-одному, пункт 1 вашего плана).
- В соответствии с `backend-only-external-api-policy.md`, генерация описаний через Yandex GPT будет реализована **исключительно на бэкенде** (внутри NestJS, без прямых вызовов с клиента). Frontend будет только запрашивать инициализацию сессии.

## 2. Сильные стороны вашего плана

- **Забота о данных пользователя:** Продуманная система модальных окон исключает случайную потерю несохраненного маршрута в Planner'е.
- **Бесшовный UX:** Четкое разделение состояний кнопок («Применить» vs «Обновить») дает пользователю понимание, что он работает с одним и тем же планом, а не плодит дубликаты.
- **Интеллектуальный онбординг:** Идея встречать пользователя в чате уже сформированным контекстом маршрута из Planner'а с призывом к действию делает ИИ по-настоящему проактивным помощником.

## 3. Решения и оптимизации

1. **Состояние "несохраненных изменений" (Dirty State):** Модальное окно при переходе из чата в Planner нужно показывать **только если** в текущем открытом в Planner маршруте есть несохраненные изменения (dirty state).
2. **Асинхронность бэкенда при генерации описаний:** Запрос к Yandex GPT для генерации описаний точек маршрута может занять время.
   _Решение:_ Чат открывается мгновенно, и пока бэкенд генерирует описания, в UI чата будет показываться **стандартный существующий скелетон загрузки** (loading state сообщений), чтобы пользователь видел, что процесс идет.
3. **Кнопки быстрых действий (Quick Actions):** Вместо текстовых подсказок внутри приветственного сообщения, пользователю будут показаны кликабельные кнопки (чипсы) с вариантами изменения маршрута.

## 4. Спроектированная реализация (Пошаговый план)

### Бэкенд (API)

1. **Эндпоинт связи:** Создать/обновить эндпоинт `POST /api/ai/sessions/:sessionId/apply`, который:
   - Создает новый `trip` (если `tripId` в сессии `null`) или обновляет существующий.
   - Возвращает `tripId`.
2. **Эндпоинт инициализации из Planner:** `POST /api/ai/sessions/from-trip/:tripId`
   - Ищет существующую `ai_session` для этого `tripId`. Если нет — создает.
   - Запускает процесс: читает точки маршрута, группирует по датам, запрашивает Yandex GPT для генерации коротких описаний точек.
   - Добавляет в историю чата приветственное сообщение и сгенерированный маршрут.

### Фронтенд (Web)

**Архитектурное решение (Draft Model vs Persisted Trip):**
Так как в Planner реализован Autosave, нам нужно четко разделять «сохраненный маршрут в БД» и «новую версию плана в чате».
Кнопка «Обновить маршрут» (которая тихо мутировала БД из чата) **удаляется**, чтобы Planner оставался единственной точкой финального редактирования и сохранения (SSOT).

1. **Zustand Stores:**
   - `isDirty` возвращается в persist-хранилище `trip.store.ts`.
   - В `ai-query.store.ts` кнопка «Применить план» создает новый маршрут (если у сессии его нет). Для последующих планов в уже привязанном чате состояние сразу переходит в «черновик» (появляется кнопка перехода в Planner), без мутации БД из чата.
2. **Компонент кнопок в чате (`MessageBubble.tsx`):**
   - `if (!session.tripId)` -> кнопка «Применить план в маршрут» -> `mutation(create) -> update session with tripId`.
   - `else (маршрут уже привязан)` -> кнопка «Применить план» скрывается, отображается только «Открыть Planner 🗺️».
3. **Навигация AI -> Planner:**
   - Переход по ссылке `/planner?applyTripId=123&draftMessageId=456`. (добавляется draftMessageId, чтобы Planner знал, какую именно новую версию из чата нужно показать/сравнить).
   - На странице Planner в `useEffect` перехватываем параметр.
   - Сравниваем `applyTripId` с текущим открытым в сторе.
   - Если открыт тот же самый маршрут (или есть `isDirty`) — показываем модалку конфликта версий.
4. **Навигация Planner -> AI:**
   - Кнопка «Редактировать с AI» всегда вызывает сохранение `saveCurrentTrip()`.
   - Отправляет запрос `/api/ai/sessions/from-trip/:tripId`.
   - Редиректит на `/ai-assistant?sessionId=...`.

## 5. Вординги и логика модалок (Унифицированный подход)

Модалка конфликта (`PlannerConflictModal`) перенесена со страницы Planner на страницы-источники переходов (AI-чат, Страница Тура, Главная). Это позволяет разрешить конфликт до смены контекста.

### 5.1. Модалка 1 (Открыта старая версия того же маршрута)

Показывается на странице `AIAssistantPage`, когда пользователь нажимает «Открыть Planner», но `messageId` текущего плана отличается от того, что уже загружено в Planner для этого же маршрута.

**Заголовок:** Внимание
**Текст:** В планировщике сейчас открыта старая версия этого же маршрута. Если вы продолжите, все текущие точки на карте будут безвозвратно заменены на новую версию.
**Кнопки:**

- `Сохранить старый и заменить` (Автосохранение текущего `currentTrip` в БД + применение нового плана)
- `Заменить без сохранения` (Тихое применение нового плана поверх текущего состояния Planner'а)
- `Перейти в планнер (посмотреть старый)` (Переход в Planner без применения нового плана)
- `Отмена` (Закрытие модалки, остаемся в чате)

### 5.2. Модалка 2 (В Planner открыт совершенно другой маршрут)

Показывается на странице `AIAssistantPage` (и аналогичная на странице готового тура), когда `openedPlannerTripId` не совпадает с `targetTripId` из чата (и это не гостевой пустой маршрут).

**Заголовок:** Внимание
**Текст:** Сейчас в Planner открыт маршрут **«[Название маршрута]»**. Если продолжить, он будет заменён выбранным маршрутом. Выберите действие:
**Кнопки:**

- `Сохранить старый и заменить` (Автосохранение `currentTrip` в БД + открытие нового маршрута из чата)
- `Заменить без сохранения` (Тихое открытие нового маршрута из чата)
- `Перейти в планнер (посмотреть старый)` (Переход в Planner для инспекции старого маршрута)
- `Отмена` (Закрытие модалки, остаемся в чате)
