# 1. Задачи для реализации в коде (NestJS Backend - `apps/api/src/ai/`)

Твоя цель — внедрить точечные мутации, векторный поиск и детерминированный пайплайн, как описано ниже:

## 1.1. Удаление Legacy и Shadow-режимов
*   **Удалить все Feature Flags (FF)**, связанные с включением/отключением нового AI-пайплайна (`FF_PLANNER_V2_ENABLED`, `FF_INTENT_ROUTER_V2`, `FF_POLICY_CALC_V2`, `FF_VECTOR_PREFILTER_REDIS`, `FF_LOGICAL_ID_FILTER_V2`, `FF_MASS_COLLECTION_V2`, `FF_DETERMINISTIC_PLANNER_V2`).

*   Удалить легаси-ветки генерации маршрута, оставив только новый pipeline.

## 1.2. Intent Router (Анализ намерений)
*   **Интеграция с GPT-4o-mini**: Переписать `IntentRouterService.ts`. Сейчас он использует regex-эвристики. Нужно использовать LLM (GPT-4o-mini) для классификации сообщений пользователя с учетом истории чата (до 10 сообщений) и списка текущих ID точек маршрута (`current_route_pois`).
*   **Промпт для Intent Router**:
    *"Ты — роутер намерений для туристического ИИ-планировщика. Проанализируй новое сообщение пользователя с учетом истории чата и текущего состояния маршрута (список текущих точек: {current_route_pois}). Оцени свою уверенность (confidence от 0.0 до 1.0). Если действие направлено на конкретное место (удалить/заменить), найди его id в списке {current_route_pois} и верни в target_poi_id. Если место не определить — верни null. Верни СТРОГО валидный JSON: {"action_type": "REMOVE_POI" | "REPLACE_POI" | "ADD_DAYS" | "APPLY_GLOBAL_FILTER" | "NEW_ROUTE", "confidence": 0.85, "target_poi_id": "poi-123" | null}. Никакого текста."*
*   **Управление потоком в Controller**: Реализовать выполнение действий на основе `action_type`. Fallback: Если `action_type !== 'NEW_ROUTE'` и `confidence < 0.7` -> перерасчет `NEW_ROUTE`.

## 1.3. Реализация точечных мутаций
*   **`REMOVE_POI`**: Найти `target_poi_id` в текущем `RoutePlan`, удалить его и пересчитать время транзита между оставшимися точками (через `SchedulerService` / Haversine).
*   **`ADD_DAYS`**: "Заморозить" текущие точки (добавить их в `usedPoiIds` в `SchedulerService`) и запустить полный пайплайн генерации только для новых дней.
*   **`REPLACE_POI`**:
    *   Удалить `target_poi_id`.
    *   Найти в кэше/БД 5 ближайших альтернатив той же категории.
    *   Проверить `working_hours` (отсечь закрытые в нужное время).
    *   Использовать YandexGPT для финального выбора (Промпт: *"Пользователь попросил заменить локацию X. Время посещения: 09:00. Вот доступные альтернативы. Учитывая контекст: '{user_persona_summary}', выбери одну лучшую и объясни выбор (1-2 предложения)."*).

## 1.4. Policy Snapshot (Шаг 1)
*   **Diff Update профиля**: Реализовать сжатый портрет пользователя (`user_persona_summary`). Новые пожелания из истории должны перезаписывать старые (дрейф предпочтений).
*   **Буфер емкости**: `required_capacity` должно быть `ceil(days * 5 * 1.2)`.

## 1.5. Vector Prefilter (Шаг 3 - Redis RediSearch)
*   **Интеграция с OpenAI Embeddings**: В `vector-prefilter.service.ts` добавить получение векторов (через `text-embedding-3-small`) для новых точек. Строка для векторизации: `[Категория] Название (Доп. теги)`.
*   **Кэширование в Redis**: Сохранять эмбеддинги в Redis Hash с префиксом `ai:poi:vec:{id}`.
*   **RediSearch (`FT.SEARCH`)**: Векторизовать `user_persona_summary`. Выполнять поиск: `FT.SEARCH idx:ai:poi "*=>[KNN 200 @embedding $query_vec AS score]"`.

## 1.6. Логический отбор ID (Шаг 4)
*   Перед отправкой в YandexGPT добавить этап фильтрации через GPT-4o-mini (передаем только ID, Name, Category).
