# План миграции истории AI-чатов на Backend

Данный документ описывает процесс перевода хранения истории диалогов с ИИ-ассистентом с локального хранилища браузера (`localStorage`) на серверную базу данных (PostgreSQL).

## 1. Текущее состояние и проблемы

На данный момент система работает следующим образом:
1. **Frontend**: Хранит историю сообщений и список сессий в Zustand store (`useAiQueryStore`), который использует middleware `persist` для сохранения данных в `localStorage`. 
2. **Backend**: В таблице `ai_sessions` PostgreSQL есть структура для хранения чатов (с колонкой `messages` типа `jsonb`). Однако бэкенд использует эту таблицу **исключительно как временное хранилище контекста**.
3. **Обрезка истории**: В `apps/api/src/ai/ai.controller.ts` при сохранении сообщений в базу вызывается `.slice(-10)`. Таким образом, в базе хранятся только последние 10 сообщений.
4. **Отсутствие API**: На бэкенде нет GET/DELETE эндпоинтов для получения истории диалогов пользователем на другом устройстве. Из-за этого не работает кроссплатформенная синхронизация чатов.

## 2. Ответы на частые вопросы

**Где и как будут храниться сообщения? Нужна ли отдельная таблица?**
Отдельная таблица `ai_messages` не нужна. Согласно `project_prd.md`, сообщения хранятся прямо внутри таблицы чатов (`ai_sessions`) в колонке типа `jsonb`. Это оптимально:
- Запрос работает быстрее (нет JOIN-ов).
- Структура сообщений (роль, контент, JSON с планом) идеально ложится в JSONB.

**Сколько сообщений из чата мы будем хранить в БД?**
Мы будем хранить **ВСЕ** сообщения чата, чтобы пользователь мог скроллить историю (как в Telegram или ChatGPT). Текущая жесткая обрезка (`.slice(-10)`) при сохранении в БД будет убрана. Обрезать историю мы будем **только** при формировании контекста для отправки в саму языковую модель (LLM), чтобы не превышать лимиты токенов.

## 3. Пошаговый план миграции

### Шаг 1. Рефакторинг Backend (NestJS)

1. **Создание сервиса `AiSessionsService`**: 
   Вынести логику работы с таблицей `aiSessions` (Drizzle ORM) из `AiController` в отдельный сервис.
2. **Исправление логики сохранения истории**:
   - В методе `saveSession` убрать `.slice(-10)` для массива `messages`. В БД должны попадать все сообщения без ограничений.
   - Ограничение контекста в 10 сообщений оставить **только** перед вызовом `orchestratorService.parseIntent()`.
3. **Новые эндпоинты в `AiController`**:
   - `GET /ai/sessions` — получение списка всех сессий текущего пользователя (возвращаем только `id`, `tripId`, `createdAt` и сгенерированный `title`).
   - `GET /ai/sessions/:id` — получение конкретной сессии со всем массивом `messages`.
   - `DELETE /ai/sessions/:id` — удаление сессии.

### Шаг 2. Интеграция API на Frontend

1. В папке `features/ai-query/api/` (или `shared/api/`) описать новые HTTP-вызовы:
   - `fetchAiSessions()`
   - `fetchAiSessionById(sessionId)`
   - `deleteAiSession(sessionId)`

### Шаг 3. Рефакторинг состояния Zustand

Изменяем `apps/web/src/features/ai-query/model/ai-query.store.ts`:
1. Убираем middleware `persist` (удаляем конфигурацию работы с `localStorage`).
2. Добавляем флаг `isSessionsLoading: boolean`.
3. Создаём экшен `loadSessions()`, который будет запрашивать список чатов с бэкенда при входе пользователя или маунте компонента.
4. Обновляем существующие экшены:
   - `switchSession(id)`: Если истории сообщений для выбранной сессии нет в памяти, сделать запрос `fetchAiSessionById(id)` перед переключением.
   - `deleteSession(id)`: Выполнять API-запрос `DELETE`, и только при успехе (или оптимистично) удалять сессию из стейта.
   - `createNewSession()`: Создание сессии на сервере можно делать отложенно (при первом сообщении) — логика `POST /ai/plan` с `session_id: null` это уже поддерживает. Главное обновить список сессий в стейте после первого ответа бэкенда.

### Шаг 4. (Опционально) Миграция локальных данных

Для пользователей, у которых уже накопились планы в браузере:
1. При первом открытии приложения, если сервер вернул пустой список чатов, проверить наличие данных в `localStorage` (ключ `ai-query-store`).
2. Если данные есть — отправить их батчем на специальный новый эндпоинт бэкенда `POST /ai/sessions/import` для сохранения.
3. После успешного ответа сервера удалить ключ `ai-query-store` из браузера.
