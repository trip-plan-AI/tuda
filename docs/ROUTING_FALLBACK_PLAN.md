# Архитектура каскадного роутинга (SaaS Fallback)

## 1. Проблема
В MVP `travel-planner` построение автомобильных маршрутов занимает слишком много времени из-за использования бесплатного публичного демо-сервера `router.project-osrm.org`. Этот сервер глобально перегружен и занижает приоритет длинных автомобильных маршрутов.
Развертывание собственного OSRM сервера для территории всей России и других стран требует аренды дорогих серверов (от 16 до 64 ГБ RAM).

## 2. Решение
Внедрение паттерна **"Каскадный роутинг" (Circuit Breaker / Fallback)** с использованием бесплатных коммерческих API.
Так как расчет маршрута на фронтенде срабатывает только в момент отпускания маркера пользователем (`onDragEnd`), нагрузка распределяется равномерно.

Мы объединяем квоты трех провайдеров, использующих формат данных `OSRM Directions API v5`, чтобы бэкенд отдавал фронтенду привычный `GeoJSON`, и клиентская часть (UI) не требовала переработки.

### Выбранные провайдеры (Контур балансировки)
1. **LocationIQ** (Приоритет 1)
   - Лимит: 5000 маршрутов/день.
   - Ограничение: 2 запроса/сек.
   - Роль: Основной сервис. Обрабатывает подавляющее большинство запросов.
2. **Geoapify** (Приоритет 2)
   - Лимит: 3000 маршрутов/день.
   - Ограничение: 5 запросов/сек.
   - Роль: Буфер для конкурентных запросов. Если 5 пользователей одновременно запросят маршрут, LocationIQ "отрежет" 3 запроса ошибкой 429, которые мгновенно подхватит Geoapify.
3. **OpenRouteService (ORS)** (Приоритет 3)
   - Лимит: 2000 маршрутов/день.
   - Ограничение: 40 запросов/мин (~0.6/сек).
   - Роль: Жесткий резерв на случай падения первых двух.
4. **Project OSRM Demo** (Приоритет 4)
   - Роль: "Последняя надежда". Медленно, но бесплатно и без лимитов.

Суммарная пропускная способность: **10 000 автомобильных маршрутов в день**.

## 3. Алгоритм работы (`apps/api/src/geosearch/geosearch.service.ts`)
1. Бэкенд получает координаты точек от фронтенда (`/geosearch/route`).
2. Метод `osrmRoute` проверяет тип профиля (`driving` или другой).
3. Пешеходные и вело-маршруты отправляются на быстрый сервер сообщества `routing.openstreetmap.de` (оставляем как есть, работает отлично).
4. Автомобильные маршруты проходят через цикл (каскад) провайдеров:
   - Делаем запрос (fetch) к `LocationIQ`.
   - Если статус `200 OK` — возвращаем данные клиенту.
   - Если статус `429 Too Many Requests` или `5xx Server Error` — перехватываем ошибку и отправляем этот же запрос в `Geoapify`.
   - Если `Geoapify` вернул ошибку — отправляем в `ORS`.
   - Если все SaaS провайдеры вернули ошибку — отправляем в `router.project-osrm.org`.
