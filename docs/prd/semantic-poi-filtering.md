# PRD: Semantic POI Filtering with Broad Provider Search

## 1. Введение и проблематика
В текущей реализации пайплайна ИИ (AI Pipeline) при формировании маршрутов для некоторых городов (например, Саратов) возникает проблема нехватки уникальных точек интереса (POI). Планировщик либо оставляет дни полупустыми (с одной точкой), либо вынужден дублировать точки или генерировать фейковые через LLM.

**Корень проблемы:**
Жесткая предварительная фильтрация по "категориям" в запросах к провайдерам данных (KudaGo, Overpass). 
Например, если LLM в запросе пользователя выделила категорию `attraction` (достопримечательность), код формировал узкий запрос к Overpass `tourism=attraction`. В OpenStreetMap реальные достопримечательности могут быть размечены как `historic=monument`, `amenity=place_of_worship` или даже просто `building=yes` с названием. Как следствие, провайдеры возвращают пустые или крошечные списки, хотя физически интересных мест в городе (и вокруг него) достаточно.

## 2. Предлагаемое решение
Отказ от жесткой фильтрации по категориям на стороне провайдеров данных (KudaGo/Overpass). Провайдеры должны возвращать **максимально широкий спектр всех потенциально интересных мест** в заданном радиусе. Затем этот "грязный" массив данных (после базовой очистки и приоритизации) передается в `semantic-filter.service.ts` (YandexGPT/OpenRouter). 

**LLM берет на себя роль единственного интеллектуального фильтра**, который:
1. Понимает контекст и предпочтения пользователя.
2. Анализирует "грязный" список мест.
3. Отбирает лучшие, наиболее подходящие локации, даже если они не вписываются в строгие классические категории.

## 3. Требования (Requirements)

### 3.1 Изменения в `overpass-client.service.ts`
- Убрать метод `buildCategoryQuery`, который генерировал узкие фильтры.
- Заменить запрос на универсальный и широкий. Искать узлы (nwr), содержащие хотя бы один из значимых для туриста тегов (например, `tourism=*`, `historic=*`, `amenity=restaurant|cafe|theatre|cinema|place_of_worship`, `leisure=park|garden`, `shop=mall`).
- Не фильтровать финальный результат по массивам `intent.categories` и `intent.excluded_categories`.
- Увеличить таймаут или оптимизировать радиус, чтобы избежать падения базы (рекомендуется динамический радиус или фиксированный ~20 км от центра для первого запроса).

### 3.2 Изменения в `kudago-client.service.ts`
- Игнорировать маппинг `intent.categories` на категории KudaGo.
- Запрашивать максимально широкий список популярных мест (places) для заданного города с сортировкой по популярности/рейтингу, чтобы получить все доступные локации.

### 3.3 Изменения в `provider-search.service.ts`
- Собирать результаты от всех провайдеров.
- **Базовая эвристика (Пре-фильтр):** Отсекать полный мусор (безымянные точки).
- **Extreme Fallback (Генерация недостающих точек через LLM):**
  - Если после выполнения всех запросов (даже с расширенным радиусом 15+ км) общее количество найденных мест оказалось меньше требуемого минимума (`days * 2`), включается спасательная логика.
  - Система формирует запрос к `LlmClientService` с просьбой сгенерировать недостающее количество реально существующих мест (достопримечательностей, парков, кафе) для заданного города, исключая те, что уже найдены.
  - **Координаты сгенерированных точек:** Так как LLM не может надежно выдавать точные GPS-координаты, сгенерированным точкам присваиваются "примерные координаты" (координаты первой найденной реальной точки + небольшое случайное смещение `~0.02` градуса), чтобы они логично группировались на карте в районе города. В будущем можно улучшить, добавив вызов геокодера (например, DaData) по названию места.
  - Событие генерации логируется и возвращается клиенту в массиве fallbacks как `LLM_GENERATED_MISSING_POIS`.
- **Сортировка:** Сортировать слитый массив по качеству данных (приоритет KudaGo, наличие картинки, рейтинг, длина описания).
- **Лимитирование для LLM:** Поскольку LLM (особенно YandexGPT Lite) имеет лимит токенов контекста (~8k), передавать весь массив из 5000 точек нельзя. `provider-search` должен обрезать отсортированный массив до **Топ-100** лучших точек и передавать их дальше по пайплайну.

### 3.4 Изменения в `semantic-filter.service.ts`
- Получать на вход расширенный массив точек (до 150 шт.).
- Формировать промпт так, чтобы минимизировать количество используемых токенов при передаче точек (передавать только `id`, `name`, `category` или базовое описание).
- Задача LLM: выбрать из предоставленного списка от `Math.max(intent.days * 2, 6)` до 15 точек, которые максимально соответствуют строке `intent.preferences_text` и другим намерениям пользователя.
- LLM должна выступать финальным судьей: она должна отсеять "лишние" категории (если юзер просил исключить храмы — LLM сама их не выберет, проанализировав названия).

### 3.5 Ограничения по питанию (anti-food-spam)
- Проблема: при широком поиске LLM может переполнять маршрут кафе и ресторанами.
- Правило по умолчанию (если пользователь явно не просил гастро-фокус):
  - Основные приемы пищи (`restaurant`): **не менее 1 и не более 3 на день**.
  - Перекусы (`cafe`): **не более 2 на день**, и только как дополнение между основными активностями.
- Явный opt-out от питания:
  - Если пользователь написал "без еды", "не нужно есть", "без питания", "без кафе", "без ресторанов", минимальная квота питания отключается.
- Гарантия минимума питания:
  - Если opt-out не задан, маршрут обязан содержать минимум **1 точку питания в день**.
  - Приоритет точки питания для гарантии: `restaurant`, если недоступен — `cafe`.
- Разделение ответственности:
  - `semantic-filter.service.ts` отвечает за **квоты и баланс категорий** в выбранном списке POI.
  - `scheduler.service.ts` отвечает за **тайминги**:
    - между `restaurant` и следующим `restaurant` минимум 4 часа;
    - `cafe` не ставится раньше чем через 1 час после последнего `restaurant`;
    - если временное правило нарушается, планировщик сдвигает время и повторно пытается вставить точку (не теряет её сразу).
- Override: если в тексте запроса есть явный гастро-интент (например, "гастро-тур", "по кофейням", "дегустация"), строгие лимиты ослабляются.

Пример (Саратов, 1 день, без гастро-запроса):
- Допустимый результат: 2 `restaurant` (обед/ужин), 1 `cafe`, 5–7 культурных/прогулочных POI.
- Недопустимый результат: 6–8 подряд `restaurant/cafe` без достопримечательностей.

Пример (Балаково, 2 дня, без запрета питания):
- Минимально допустимо: хотя бы 1 `restaurant` или `cafe` в каждом дне.
- Недопустимо: день без единой food-точки.

### 3.6 Запрет дублирования POI между днями
- Проблема: после добавления food-ограничений в edge-case сценариях появлялось дублирование одних и тех же точек в разных днях.
- Решение:
  - в `scheduler.service.ts` используется реестр `usedPoiIds`, который запрещает повторное использование уже поставленной точки;
  - fallback-ветки (добавление food/non-food при дисбалансе дня) выбирают только неиспользованные POI;
  - убрана стратегия искусственного дублирования POI для добора до `days * 2`.
- Ожидаемое поведение:
  - если уникальных точек достаточно — повторов между днями нет;
  - если уникальных точек недостаточно — маршрут честно показывает нехватку уникальных мест, без повторного вставления тех же POI.

## 4. Ожидаемый результат и метрики успеха
- **Решение проблемы пустых маршрутов:** В 99% случаев для городов от 100 000 человек (таких как Саратов) будет найдено достаточно точек для формирования маршрута (без дублей и фейковых генераций).
- **Улучшение релевантности:** Маршруты станут более креативными, так как LLM сможет выбирать нетривиальные локации (например, арт-объекты или видовые площадки), которые ранее отсекались строгими фильтрами.
- **Метрика:** В `scheduler.service.ts` больше не должно срабатывать предупреждение о нехватке точек (кроме экстремальных случаев, вроде "тур в тайгу").

## 6. Актуализация: причины деградации и план рефакторинга v2

### 6.1 Наблюдаемые проблемы
- Дни могут формироваться с перекосом: либо только питание, либо совсем без питания.
- Иногда второй день остается почти пустым, несмотря на наличие кандидатов.
- После анти-спам ограничений качество разнообразия стало нестабильным между одинаковыми запросами.

### 6.2 Корневые причины
- В `scheduler.service.ts` основной алгоритм остается линейным по одному массиву, что плохо сочетается с жесткими food-правилами и ограничениями времени.
- Fallback-логика вмешивается в конец дня, а не в процесс планирования слотов, из-за чего при нехватке времени точки не успевают корректно распределиться.
- В `provider-search.service.ts` отбор топа делается по рейтингу без учета квот разнообразия (non-food/food balance).
- В `semantic-filter.service.ts` ограничения по еде заданы как промпт-правила, но без детерминированной валидации структуры ответа по ролям точек.

### 6.3 Целевая стратегия v2
1. Перейти на slot-based планирование дня в `scheduler.service.ts`:
   - сначала обязательный минимум non-food (если доступен),
   - затем обязательный минимум food (если пользователь не отключил питание),
   - затем заполнение оставшихся слотов с проверкой разрывов времени.
2. Разделить кандидатов на корзины до расписания: `nonFood`, `restaurants`, `cafes`.
3. Ввести строгую пост-валидацию дня:
   - минимум 2 точки/день,
   - минимум 1 non-food (если есть в пуле),
   - минимум 1 food (если нет food opt-out и food доступен).
4. Убрать любые механики повторного использования POI между днями, пока есть уникальные точки.
5. Добавить деградационный режим: если constraints не выполняются из-за малого пула, возвращать прозрачную заметку в `notes` вместо скрытого ухудшения расписания.

## 7. Актуализация v4: Квотирование пре-фильтра и защита от пустых дней
### 7.1 Наблюдаемые проблемы (Итерация 4)
- **Исчезновение достопримечательностей:** Рейтинг кафе/ресторанов (особенно из KudaGo) в среднем выше (4.5), чем у парков и музеев из Overpass (4.0). Из-за этого при формировании общего Топ-100 в `provider-search` еда вытесняет достопримечательности, и LLM получает на вход список, состоящий на 90% из еды.
- **Жадный шедулер:** При наличии 3-х не-food точек на 2 дня поездки, планировщик ставил все 3 точки в первый день (пока есть время), оставляя второй день состоять только из кафе и ресторанов.

### 7.2 Целевое решение v4 (Фаза "Жесткий баланс")
1. **Квотирование перед LLM (`provider-search.service.ts`):**
   - Перестаем сортировать все найденные POI в одну кучу.
   - Разделяем пул на `food` (restaurant/cafe) и `non-food`.
   - Оставляем Топ-50 лучших `non-food` и Топ-50 лучших `food`. Это гарантирует, что музеи и парки 100% дойдут до нейросети.
2. **Защита от лени LLM (`semantic-filter.service.ts`):**
   - Добавить жесткое требование вернуть *максимально возможное* количество точек (вплоть до `maxPlaces`), чтобы нейросеть не обрезала результат по своему желанию.
3. **Строгое резервирование по дням (`scheduler.service.ts`):**
   - Внедрить **дневной лимит (квоту)** на расходование `non-food` точек.
   - Если есть 4 `non-food` точки и 2 дня, День 1 имеет право забрать **только 2 точки**. Даже если у него осталось свободное время, он завершает активности и оставляет оставшиеся 2 точки для Дня 2.
   - Это гарантирует, что второй день никогда не останется без достопримечательностей.

### 7.3 План разработки v4
- Шаг A: рефактор `scheduler.service.ts` на slot-based алгоритм и детерминированные квоты дня.
- Шаг B: улучшить prefilter в `provider-search.service.ts` (diversity-aware top).
- Шаг C: усилить контракт `semantic-filter.service.ts` (квоты и role-aware контроль).
- Шаг D: добавить тест-кейсы качества (Балаково 2 дня, малый город, гастро-тур, без еды).

### 6.5 Критерии приемки
- Для Балаково на 2 дня оба дня заполнены, нет пустого второго дня.
- Нет дублей POI между днями при наличии достаточного числа уникальных точек.
- Нет дней «только еда», если доступны non-food точки.
- Нет дней «без еды», если пользователь не отключал питание и есть food кандидаты.

## 7. Актуализация v3: Динамические квоты и Гео-оптимизация
### 7.1 Наблюдаемые проблемы
- При многодневных маршрутах LLM возвращала мало точек (жесткий лимит до 15), из-за чего на второй/третий день просто не оставалось кандидатов.
- Планировщик использовал жесткое время транзита (25 минут), что делало маршрут разреженным.
- Точки брались в случайном порядке, что заставляло бы пользователя метаться по городу.

### 7.2 Целевое решение v3
1. **Динамическая квота от LLM (`semantic-filter.service.ts`):**
   - Запрашиваем у LLM количество точек, пропорциональное дням поездки.
   - `minPlaces = intent.days * 4`, `maxPlaces = intent.days * 6` (в среднем 5+ точек на день).
2. **Умный порядок обхода (Nearest Neighbor в `scheduler.service.ts`):**
   - Выбор следующей точки происходит не по индексу массива, а по минимальному расстоянию от предыдущей точки (по координатам `lat/lon`).
3. **Расчет реального транзита (Haversine):**
   - Вместо жестких 25 минут, время в пути вычисляется динамически на основе расстояния между точками:
     - Скорость пешехода (до 2 км): ~5 км/ч.
     - Скорость авто (более 2 км): ~25 км/ч.
4. **Квотирование по дням:**
   - Не позволять первому дню "высасывать" все достопримечательности. Равномерно делить доступный пул `non-food` точек на количество дней поездки.
